# 专家路演投屏会话设计

## 目标

将“现场投屏展示”和“管理员评审管理/导出留档”拆成两个独立职责：

- 管理员端负责创建路演评审、选择项目、选择实名专家、设置规则、监督提交状态、导出明细。
- 投屏端负责全屏只读展示当前路演项目的匿名专家评分进度和最终得分。
- 专家端只负责在授权时间内提交 0.00 到 100.00 的分数。

不新增长期“投屏账号”。投屏入口由管理员在路演评审轮次中生成临时只读链接。

## 核心原则

1. 投屏链接只在“项目路演”评审类型中生成，网络评审不生成投屏链接。
2. 投屏链接有效期与评审轮次时间区间一致：开始时间前不可用，截止时间后自动失效。
3. 投屏端不显示专家真实姓名、账号、邮箱、分组或任何身份信息。
4. 投屏端专家席位固定显示为“专家 1”“专家 2”到“专家 N”，头像统一。
5. 倒计时只用于现场节奏提示，不参与成绩计算。
6. 倒计时结束但仍有专家未提交时，不自动归零、不自动出分，只显示“已超时，等待提交”。
7. 只有全部有效专家席位提交后，才按规则计算最终分。
8. 如专家临时退出，管理员可作废该专家席位；作废后该席位不参与“全部提交”判断和最终计算。

## 用户角色与入口

### 管理员端

入口仍在“专家评审”页面。

管理员在创建或编辑路演评审时配置：

- 评审轮次和项目。
- 参评专家实名列表。
- 专家匿名席位顺序。
- 评审开始时间和截止时间。
- 倒计时时长，默认 60 秒。
- 计分规则：去最高分数量、去最低分数量。

管理员可执行：

- 生成投屏链接。
- 复制投屏链接。
- 开始当前项目评分。
- 重置当前项目倒计时。
- 查看实名专家提交状态。
- 作废专家席位。
- 查看最终分。
- 导出评分明细。

### 投屏端

入口为临时只读链接，例如：

`/review-screen/session/{sessionId}?token={signedToken}`

页面全屏展示，不显示后台导航、用户信息、系统管理按钮。

投屏端只展示：

- 当前评审轮次。
- 当前项目名称。
- 倒计时。
- 匿名专家席位。
- 各席位提交状态和分数。
- 最终得分。
- 计分规则说明。

### 专家端

专家继续使用现有专家评审入口提交路演分数。

专家提交后：

- 分数立即锁定。
- 不可修改。
- 不可撤回。
- 分数保留两位小数。
- 提交时间记录到后台明细。

## 数据模型建议

### ReviewDisplaySession

用于表示一次投屏会话。

字段建议：

- `id`
- `reviewPackageId`
- `stageId`
- `projectMaterialSubmissionId`
- `status`: `waiting` / `scoring` / `revealed` / `closed`
- `tokenHash`
- `tokenExpiresAt`
- `countdownSeconds`
- `startedAt`
- `endedAt`
- `createdById`
- `createdAt`
- `updatedAt`
- `dropHighestCount`
- `dropLowestCount`

### ReviewDisplaySeat

用于绑定实名专家和匿名席位。

字段建议：

- `id`
- `sessionId`
- `expertUserId`
- `seatNo`
- `displayName`: 例如“专家 1”
- `status`: `pending` / `submitted` / `voided`
- `voidedAt`
- `voidedById`
- `voidReason`
- `createdAt`
- `updatedAt`

### 现有 ExpertReviewScore

继续作为专家真实评分记录来源，不另建重复分数表。

投屏端读取时只返回匿名席位和分数，不返回 `expertUserId` 或专家姓名。

## 权限规则

### 管理员

系统管理员和校级管理员可以：

- 创建路演投屏会话。
- 查看实名专家和提交时间。
- 作废专家席位。
- 导出完整评分明细。

### 投屏链接

投屏链接不使用普通账号登录，但必须校验签名 token。

校验规则：

- token 必须有效。
- token 必须属于指定 session。
- 当前时间必须在评审开始和截止时间之间。
- session 未关闭。

接口返回必须脱敏：

- 不返回专家真实姓名。
- 不返回账号、邮箱、手机号。
- 不返回后台操作权限。

### 专家

专家只能提交分配给自己的路演评分。

专家不能：

- 访问投屏管理接口。
- 看到其他专家分数。
- 看到最终计算明细。
- 修改已提交分数。

## 实时同步方案

第一版采用 1 秒轮询，保证稳定性和部署兼容性。

投屏端每秒请求：

`GET /api/review-screen/sessions/{sessionId}?token={signedToken}`

返回示例：

```json
{
  "status": "scoring",
  "roundName": "第一轮路演评审",
  "projectName": "智在必行",
  "countdownSeconds": 60,
  "remainingSeconds": 18,
  "isOvertime": false,
  "seats": [
    { "seatNo": 1, "displayName": "专家 1", "status": "submitted", "score": "86.50" },
    { "seatNo": 2, "displayName": "专家 2", "status": "pending", "score": null }
  ],
  "finalScore": null
}
```

倒计时结束但未全部提交时返回：

```json
{
  "status": "scoring",
  "remainingSeconds": 0,
  "isOvertime": true,
  "message": "已超时，等待专家提交"
}
```

全部有效席位提交后返回：

```json
{
  "status": "revealed",
  "finalScore": "87.36",
  "ruleText": "去掉 1 个最高分，去掉 1 个最低分",
  "includedCount": 5
}
```

后续如现场需要更低延迟，可将轮询替换为 SSE；接口响应结构保持不变。

## 计分规则

分数范围：

- 0.00 到 100.00。
- 统一保留两位小数。
- 后端按整数分存储，例如 86.50 存为 8650。

最终分计算：

1. 取全部 `submitted` 且未 `voided` 的专家分数。
2. 按分数升序排序。
3. 去掉指定数量最低分。
4. 去掉指定数量最高分。
5. 对剩余分数取平均。
6. 最终分四舍五入保留两位小数。

规则校验：

- `dropHighestCount >= 0`
- `dropLowestCount >= 0`
- `dropHighestCount + dropLowestCount < 有效专家席位数`
- 至少保留 1 个分数参与计算。

建议 UI 中提示管理员：去分规则设置过大时不能保存。

## 管理员导出明细

导出内容应包含：

- 评审轮次。
- 项目名称。
- 专家真实姓名。
- 匿名席位编号。
- 原始分数。
- 提交时间。
- 席位状态。
- 是否作废。
- 是否被去掉最高分。
- 是否被去掉最低分。
- 是否计入最终分。
- 最终得分。
- 计分规则。

导出用于赛后留档，不用于投屏端展示。

## 投屏页面视觉要求

投屏页应为独立全屏视觉，不沿用后台壳。

建议结构：

- 顶部：赛事名称、评审轮次、当前项目、倒计时。
- 中部：专家席位网格。
- 底部：提交进度、计分规则简述。
- 结果态：居中显示最终得分，背景有克制动效。

匿名席位状态：

- `pending`: 灰色等待态。
- `submitted`: 蓝色或绿色高亮并显示分数。
- `voided`: 低透明度显示“席位作废”。
- `overtime pending`: 倒计时为 0 后，仍显示等待提交，不出分。

### 动画与过渡

投屏端需要有丝滑、克制的状态过渡，服务于现场观看和节奏感，不做炫技动画。

必须覆盖：

- 进入评分页时，专家席位按顺序轻微上浮淡入，整体时长控制在 400ms 到 600ms。
- 专家提交后，对应席位从等待态平滑切换到已提交态，分数使用 250ms 到 350ms 的淡入和轻微缩放。
- 倒计时进入最后 10 秒时，只做颜色和轻微脉冲提示，不做大幅闪烁。
- 全部有效专家提交后，先停留 600ms 确认所有席位状态，再切换到最终得分页。
- 最终得分页使用居中分数放大淡入，时长控制在 700ms 内。

禁止：

- 大面积闪烁。
- 长时间 loading 动画。
- 复杂 3D 翻转。
- 影响读数稳定性的跳动数字。
- 因动画阻塞实时数据刷新。

技术原则：

- 过渡动画只在前端表现层处理。
- 分数、倒计时和最终结果以后端返回为准。
- 轮询更新期间如果新数据到达，应优先更新数据，再播放轻量状态过渡。
- 用户刷新页面后，页面根据后端状态直接恢复，不依赖本地动画状态。

## 稳定性要求

1. 投屏链接 token 与 session 绑定，不能复用到其他 session。
2. 投屏端接口只读，不能修改任何数据。
3. 轮询失败时页面保留上一次成功数据，并显示“连接恢复中”。
4. 连续失败恢复后自动同步最新状态。
5. session 状态和分数计算均以后端为准，前端只展示。
6. 浏览器刷新后仍能恢复当前 session 状态。
7. 动画不能阻塞轮询和状态恢复；任何动画失败都不能影响数据展示。

## 不做内容

第一版不做：

- WebSocket。
- 专家真实身份投屏展示。
- 投屏账号登录。
- 未提交专家自动按 0 分。
- 倒计时结束自动出分。
- 网络评审投屏。
- 现场弹幕或复杂大屏动画。

## 验收标准

1. 管理员只有在项目路演评审中才能生成投屏链接。
2. 投屏链接有效期与评审轮次开始和截止时间一致。
3. 投屏端只显示专家 1 到专家 N，不显示真实姓名。
4. 专家提交后，大屏 1 秒内看到对应匿名席位分数。
5. 60 秒倒计时结束但有人未提交时，大屏不出最终分。
6. 所有有效专家提交后，大屏自动显示最终得分。
7. 管理员作废专家席位后，该席位不参与最终分计算。
8. 管理员可导出实名评分明细。
9. 专家不能看到其他专家分数。
10. 投屏端刷新页面后仍能恢复当前状态。
11. 投屏端状态切换和最终出分有丝滑过渡，但不影响读数清晰度和实时同步。
